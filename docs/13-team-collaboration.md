# 团队协作规范与角色分工

## 一、团队构成

费用单管理系统采用小团队协作模式（3-5 人），由 AI 主导编码、人类把关质量与交付。

| 角色 | 承担方 | 职责 |
|------|--------|------|
| 架构设计 | Claude | 架构、技术选型、数据模型、接口契约、文档 |
| 后端开发 | Claude | FastAPI API、ORM 模型、业务逻辑、渲染引擎 |
| 前端开发 | Claude | Vue3 页面、组件、状态管理、模板渲染 |
| 产品/需求 | 人类成员 | 需求管理、验收、业务规则确认、需求变更对接 |
| 技术评审 / Tech Lead | 人类成员 | 代码评审、技术决策参与、质量把关 |
| 测试 / QA | 人类成员 | 功能测试、权限场景验证、Bug 反馈 |
| 运维 / 部署 | 人类成员 | 环境搭建、部署上线、数据库运维 |

## 二、协作流程

```
产品负责人 ──提需求──▶ Claude(架构+编码)
                          │
                          ├─ 产出代码 + 实现说明
                          ├─ 产出接口契约 (OpenAPI)
                          └─ 产出 QA 验证点清单
                          │
        ┌─────────────────┼─────────────────┐
        ▼                 ▼                 ▼
   Tech Lead 评审    QA 功能测试      运维部署验证
        │                 │                 │
        └────── 反馈问题 ──┴──── Claude 修复 ─┘
                          │
                    里程碑评审 (M1-M4)
```

## 三、Claude 的交付物标准

每完成一个模块，我会产出以下三类交付物，方便人类成员各司其职：

### 3.1 给 Tech Lead（评审用）
- 代码变更说明：改了哪些文件、核心逻辑、设计取舍
- 接口契约：FastAPI 自动生成的 OpenAPI 文档（/docs）
- 关键决策点：需要 Tech Lead 拍板的技术选择

### 3.2 给 QA（测试用）
- 验证点清单：每个功能的正常路径 + 边界场景
- 权限测试矩阵：admin / user 在各机构层级下的预期行为
- 测试数据准备说明：如何造数据、初始账号

### 3.3 给运维（部署用）
- 环境依赖说明：Python/Node 版本、依赖清单
- 配置项说明：.env 各字段含义
- 部署步骤：迁移命令、启动命令

## 四、里程碑评审节点

每个里程碑作为正式评审节点，人类团队集中验收：

| 里程碑 | 时间 | 评审内容 | 参与角色 |
|--------|------|----------|----------|
| M1 | W1 结束 | 登录 + 机构树展示 | Tech Lead + QA |
| M2 | W3 结束 | 字典/模板/公章维护 + 模板预览 | 全员 |
| M3 | W4 结束 | 单据填报 + PDF/Excel 导出 | 全员 |
| M4 | W5 结束 | 一期交付，全流程验收 | 全员 + 产品 |

## 五、需求变更流程

涉及以下变更时，由产品负责人发起，我评估影响后再实施：
- 新增/修改数据库字段
- 调整权限规则
- 修改模板结构
- 新增功能模块

变更影响评估包括：受影响的文档、代码模块、数据库迁移、工期影响。

## 六、构建顺序

严格按 [07-development-plan.md](07-development-plan.md) 的排期推进，不跳阶段：

```
W1: 后端骨架 → 建表 → 认证 → 前端骨架 → 登录页
W2: 机构树 CRUD → 用户管理
W3: 费用字典 → 公章 → 默认配置 → 模板系统
W4: 单据 CRUD → 填报表单 → 列表详情 → PDF/Excel 导出
W5: 联调 → 权限测试 → 打印 → Bug 修复 → 部署文档
```

## 七、代码规范

### 后端 (Python)
- 遵循 PEP 8，使用 Ruff 检查
- 类型注解完整，Pydantic 校验所有出入参
- 业务逻辑放 services 层，路由层只做编排
- 关键路径写 pytest 测试

### 前端 (Vue3 + TS)
- Composition API + `<script setup>`
- TypeScript 严格模式，类型定义放 types/
- API 调用统一走 api/ 封装层
- ESLint + Prettier 格式化
