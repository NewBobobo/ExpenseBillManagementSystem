# 多 AI 协作中枢

本目录是三个 AI 角色之间的「交接班」通道。我们**无法实时通话**：每个会话只在主动读取本目录文件时才能看到他人的更新。因此所有协作必须落到文件，遵守下面的纪律。

## 一、角色

| 代号 | 工具 | 职责 | 只写区域 |
|------|------|------|----------|
| ARCH | Claude (Kiro) | 架构、数据模型、接口契约、评审、拆任务 | `CONTRACTS.md` 全部；`BOARD.md` 派活 |
| FE | CodeWhy | 前端 Vue3 实现 | `frontend/**`；BOARD/LOG 自己的行 |
| BE | AwerCoder | 后端 FastAPI 实现 | `backend/**`；BOARD/LOG 自己的行 |

## 二、协作文件

- [BOARD.md](BOARD.md) — **任务看板**：当前每个人在做什么、状态、阻塞。开工先看这里。
- [LOG.md](LOG.md) — **工作流水日志**：每次有进展就追加一行，留下时间痕迹，**只追加不修改历史**。
- [CONTRACTS.md](CONTRACTS.md) — **接口契约**：前后端对齐的唯一事实来源，只有 ARCH 能改契约本身。

## 三、铁律（每个会话开工/收工必做）

**开工三件事**
1. 读 `BOARD.md` → 找到标着自己代号、状态为 `TODO` 或 `BLOCKED→已解` 的任务。
2. 读 `LOG.md` 末尾 10 行 → 了解别人刚做了什么。
3. 把自己要做的任务在 BOARD 上改成 `DOING`，并在 LOG 追加一行「开始 XXX」（时间格式 `YYYY-MM-DD HH:MM:SS`）。

**收工三件事**
1. 在 LOG 追加一行：做完了什么、改了哪些文件、有没有给别人留活（时间格式 `YYYY-MM-DD HH:MM:SS`）。
2. 更新 BOARD 自己那行的状态（`DONE` / `BLOCKED` / `REVIEW`）和最后更新时间（`YYYY-MM-DD HH:MM:SS`）。
3. 若产生需要对方配合的事（如前端需要某接口），在 BOARD 给对方**新建一行任务**并 @ 对方代号。

**时间格式规范**：所有协作文件（LOG/BOARD/CONTRACTS）中的时间戳统一为 `YYYY-MM-DD HH:MM:SS`（精确到秒），便于追溯和排序。

## 四、冲突规避

- 各自只改「只写区域」的文件，避免同时编辑同一文件。
- BOARD/LOG 是共享文件：**只改/追加自己代号的行**，不要动别人的行。
- 接口要变更：FE/BE 不直接改 `CONTRACTS.md`，而是在 BOARD 给 ARCH 开一条 `@ARCH 契约变更申请`，由 ARCH 裁定后修改。

## 五、行号约定（写日志/任务时引用）

引用代码用 `相对路径:行号`，例如 `backend/app/api/auth.py:42`，方便对方点击定位。

## 六、会话启动咒语（上下文满了重开必用）

AI 会话的对话上下文是**临时的、会丢的**；但本目录文件 + 代码是**持久的**。
所以"重开会话"不是从头来，而是"换个人接班看交接文档"。任一会话被新开时，
把对应咒语贴给它即可在 3 分钟内恢复"我是谁/做到哪/下一步"：

**架构师会话(ARCH)**
```
你是本项目架构师会话，代号 ARCH。先读 docs/collab/README.md 与 BOARD.md、
LOG.md 末尾15行、CONTRACTS.md，了解当前进度。你的职责是统筹进度、锁接口契约、
派活、评审，不亲自写业务代码。继续未完成的 ARCH 任务。
```

**前端会话(FE / CodeWhy)**
```
你是本项目前端开发会话，代号 FE。先读 docs/collab/README.md(协作纪律)、
BOARD.md(找标 FE、状态 DOING/TODO 的任务)、LOG.md 末尾10行、CONTRACTS.md(接口字段)。
按"开工三件事"接班，只改 frontend/** 和 BOARD/LOG 里自己的行。继续做到哪算哪。
```

**后端会话(BE / AwerCoder)**
```
你是本项目后端开发会话，代号 BE。先读 docs/collab/README.md(协作纪律)、
BOARD.md(找标 BE、状态 DOING/TODO 的任务)、LOG.md 末尾10行、CONTRACTS.md(接口字段)。
后端虚拟环境在 backend/venv(Python3.11)。按"开工三件事"接班，只改 backend/**
和 BOARD/LOG 里自己的行。继续做到哪算哪。
```

## 七、防止"重头开始"的纪律（降低重开代价）

1. **任务切小**：一次只领一个能在单个会话内做完的小任务（如"只做登录接口"），
   做完即收工，别憋大功能——任务越小越不容易撞上下文上限。
2. **DOING 写到能接班**：收工时若任务没做完，LOG 必须写清
   「做到了 `某文件:行号` 的第几步，下一步具体要做什么」，颗粒度细到接班人不用猜。
3. **先写 LOG 再关会话**：上下文快满时，先完成"收工三件事"再开新会话，
   绝不在没留痕的情况下丢弃会话。
